# 제품 요구사항 : BotC 오프라인 플레이 보조 웹앱

## 1. 프로젝트 개요
* **프로젝트명:** BotC Offline Companion (가칭)
* **목표:** 오프라인 보드게임 'Blood on the Clocktower'의 진행을 돕는 모바일 중심(Mobile-First) 웹 애플리케이션 구축. 스토리텔러(ST)의 마도서 관리 부담을 줄이고, 플레이어의 야간 비동기 행동 및 주간 실시간 동시 투표를 디지털화함.
* **핵심 가치:** 빠른 접근성(익명 로그인), 직관적인 상태 동기화, 제로 트러스트(Zero-Trust) 기반의 완벽한 정보 스니핑 차단.

## 2. 기술 스택 및 아키텍처 제약
### 2.1. 기술 스택
* **프론트엔드:** Vite, React 19, TypeScript, Tailwind CSS
* **상태 관리:** Zustand (`useGameStore` - 공개 상태 / `useSecretStore` - ST 전용 상태)
* **백엔드/데이터베이스:** Firebase Auth(익명 로그인), Firebase Realtime Database
* **호스팅:** Github Pages (actions - deploy)

### 2.2. 아키텍처 제약 사항
1. **절차적 DOM 조작 금지:** `document.getElementById` 등 사용을 엄격히 금지하며, 상태(State) 기반 렌더링만 허용.
2. **독립된 UI 캡슐화:** 모든 UI는 재사용 가능하고 부수 효과(Side-effect)가 없는 독립된 React 컴포넌트로 분리.
3. **보안 최우선 설계:** 클라이언트에서의 데이터 조작 및 스니핑을 원천 차단하기 위해 데이터베이스 루트 레벨에서 `public`과 `secret` 권한을 물리적으로 분리.
4. **세션 복원력:** 브라우저 새로고침이나 네트워크 단절 시에도 `localStorage`와 Firebase Auth를 통해 즉시 기존 상태와 좌석으로 복구 가능해야 함.

## 3. 핵심 사용자 여정 및 페이즈 (Game Flow)

### 3.1. Phase 1-A: 입장 관리 및 대기 (Lobby)
* **목적:** 참가자 확인 및 좌석 배치.
* **ST 권한:** 방 생성, 참여 코드 표시, 접속자 모니터링. 오프라인 좌석에 맞춘 플레이어 자리 배치 확정(수동 드래그 앤 드롭 또는 랜덤 셔플).
* **플레이어 뷰:** 닉네임 입력 후 입장. 타 플레이어 접속 목록 확인.
* **상태 전환:** ST가 자리 배치를 확정하고 '세팅 단계'로 이동.

### 3.2. Phase 1-B: 마도서 세팅 및 직업 배정 (Setup)
* **목적:** 게임의 판(Grimoire) 설계.
* **ST 권한:** 확정된 인원에 따른 진영 비율(마을주민/외부자/미니언/악마) 확인. 이번 게임에 등장할 직업 및 악마의 블러프(가짜 직업 3개) 선택. 플레이어별 직업 할당.
* **플레이어 뷰:** "스토리텔러가 마도서를 세팅 중입니다." 대기 화면.
* **상태 전환:** ST가 세팅을 완료하고 '게임 시작'을 누르면 각 기기로 비밀 정보가 전송되며 1일 차 밤으로 진입.

### 3.3. Phase 2: 야간 페이즈 (The Night)
* **목적:** 비동기 능력 사용 및 ST의 일괄 판정.
* **플레이어 권한 (비동기):** 기상 순서를 기다릴 필요 없이 각자의 기기에서 즉시 능력 행동 입력 (예: 대상 선택).
* **ST 처리 로직:** 플레이어들의 행동이 `secret` 노드에 수집됨. ST는 마도서 뷰에서 공식 기상 순서대로 정렬된 대기열(Queue)을 확인하며, 독/술 취함 상태 등을 고려해 개별 결과값을 순차적으로 확정 및 승인.
* **상태 전환:** 처리가 끝나고 ST가 '낮으로 전환' 버튼 클릭 시, 아침이 밝으며 각 플레이어 기기에 능력 결과가 일괄 통보됨.

### 3.4. Phase 3: 주간 페이즈 및 투표 (The Day & Voting)
* **목적:** 공개 토론, 지목, 실시간 동시 투표.
* **동시 투표 시스템:** 지목(Nomination) 발생 시 투표 개시. 모든 플레이어가 찬성/반대를 입력하며, 전체 화면에 누가 찬성했는지 **실시간(Real-time)으로 표시됨**. 과반수(Majority) 달성 여부 시각화.
* **유령 투표권:** 사망자가 투표 시도 시 경고창 노출. 투표 참여 시 즉시 해당 유령의 투표권 데이터(`hasGhostVote`) 소모 처리.

### 3.5. Phase 4: 처형 및 종료 판정 (Execution & End)
* **자동 처형:** 낮 동안 누적된 투표 결과를 바탕으로, '밤으로 전환' 시 가장 표를 많이 받은 대상(동률 규칙 포함)을 시스템이 자동 사망 처리.
* **게임 종료:** ST가 승리 진영(Good/Evil) 선언 시, `secret` 노드의 모든 정보가 `public`으로 복사되며 전원이 마도서의 진실을 확인할 수 있는 복기 모드 오픈.

## 4. 정보 가시성 스펙 (Information Visibility)

정보의 불확실성을 유지하기 위해 데이터는 세 가지 뷰로 엄격히 통제됩니다.

### 4.1. 스토리텔러 마도서 뷰 (절대적 진실)
* **권한:** ST (모든 `secret` 데이터 열람 및 수정)
* **표시 정보:**
    * 모든 플레이어의 실제 직업과 진영.
    * 숨겨진 상태 이상 (독에 걸림, 자신이 다른 직업이라고 착각하는 미치광이/술 취한 자 등).
    * 플레이어가 입력한 야간 행동 원본 데이터와 ST가 전송할 조작된 결과.

### 4.2. 플레이어 개인 뷰 (개인용 가공 데이터)
* **권한:** 해당 플레이어 본인 (Firebase Rules로 본인 UID 데이터만 읽기 허용)
* **표시 정보:**
    * 시스템이(또는 ST가) 부여한 본인의 직업과 진영 (실제 진실과 다를 수 있음).
    * ST가 확정하여 보내준 야간 행동 결과 메시지.
    * 악마일 경우 블러프(가짜 직업) 정보 3개.
* **제한 사항:** 본인의 상태 이상 여부는 절대 표시되지 않음. 타인의 화면 데이터 열람 불가.

### 4.3. 마을 광장 뷰 (Public 팩트)
* **권한:** 방에 접속한 모든 사용자
* **표시 정보:**
    * 페이즈 정보 (예: 2일 차 낮).
    * 생존자 및 사망자 현황.
    * 사망자의 **유령 투표권(Ghost Vote)** 잔여 여부 (시각적 아이콘).
    * 현재 진행 중인 투표의 실시간 찬성/반대 현황 및 지목 대상.
* **제한 사항:** 직업이나 진영에 대한 단서는 일절 포함되지 않음.
